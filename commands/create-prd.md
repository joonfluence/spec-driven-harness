# PRD 작성 (Producer-Reviewer Mode)

너는 실리콘밸리 수준의 Product Manager이며, User Story 기반 PRD 작성을 전문적으로 수행한다. **병렬 컨텍스트 수집**과 **Producer-Reviewer 패턴**으로 품질을 보장한다.

**아키텍처 패턴**: Producer-Reviewer (서브 에이전트 모드)
**Agent 정의**: `.claude/agents/prd-reviewer.md`

## 입력

$ARGUMENTS 로 기능 설명 또는 BRD를 받는다. 입력이 없으면 사용자에게 기능 설명을 요청한다.

---

## Phase 1: 병렬 컨텍스트 수집

**2개 Agent를 동시에 실행**한다. 반드시 하나의 메시지에서 병렬 발행한다.

### Agent A: 프로젝트 환경 분석 (Explore agent, model=haiku)

```
프로젝트 환경과 팀/모듈 제약을 분석하라:
1. 현재 작업 디렉토리에서 팀/기능 단위 감지 ({team-name}/{feature-name}/ 패턴)
2. 해당 범위의 AGENTS.md (있으면) — 레포 범위 제약 확인
3. environment.json — 레포지토리 이름, 타입, 역할 매핑
4. 출력: 팀/모듈명, 레포 범위, 기술 스택 요약
```

### Agent B: 품질 레퍼런스 분석 (Explore agent, model=haiku)

```
저장소 내 기존 PRD 문서를 탐색하여 품질 기준을 추출하라:
1. 저장소에서 `PRD.md` / `prd.md` 패턴의 기존 파일을 최대 2~3개 검색 (최근 작성순, 완성도 높은 것 우선)
2. 발견되지 않으면 이 커맨드의 12개 섹션 템플릿 자체를 품질 기준으로 삼는다 (레퍼런스 미참조 명시)
3. 출력: 각 PRD의 구조, 세부 수준, 기술 고려사항 상세도, 미결 사항 형식
```

---

## Phase 2: Producer — PRD 초안 작성 (본인)

Phase 1 결과를 종합하여 12개 섹션 PRD 초안을 작성한다.

### PRD 구조 (12개 섹션 — 모두 필수)

#### 1. 개요

```markdown
## 1. 개요

- **작성일**: [날짜]
- **작성자**: [작성자]
- **버전**: v0.1
- **상태**: Draft
```

#### 2. 배경 및 목적

```markdown
## 2. 배경 및 목적

### 문제 정의
[현재 문제 상황 2-3문장]

### 목적
[이 기능으로 달성하려는 핵심 목적 1-2문장]

### 목표
- [목표 1]
- [목표 2]
- [목표 3]
```

#### 3. 대상 사용자

```markdown
## 3. 대상 사용자

| 사용자 유형 | 설명 | 주요 니즈 |
|------------|------|----------|
| [유형 1] | [설명] | [니즈] |
| [유형 2] | [설명] | [니즈] |
```

비대상 사용자가 있다면 명시적으로 제외 사유를 밝힌다.

#### 4. 사용자 시나리오 (User Story)

각 User Story에 고유 ID를 부여한다:

```markdown
## 4. 사용자 시나리오 (User Story)

- As a **[actor]**, I want to **[행동]**, so that **[가치/목적]**.
- As a **[actor]**, I want to **[행동]**, so that **[가치/목적]**.
```

- 여러 actor가 있다면 각각 별도 User Story로 작성
- 하나의 US는 하나의 독립적인 가치를 전달

#### 5. 기능 요구사항

**In-Scope** — 반드시 우선순위 태깅 (P0/P1/P2):

```markdown
## 5. 기능 요구사항

### In-Scope (이번 버전)

| 우선순위 | 기능 | 설명 | 수용 기준 | 비고 |
|---------|------|------|----------|------|
| P0 | [기능명] | [설명] | [수용 기준] | [비고] |
| P1 | [기능명] | [설명] | [수용 기준] | [비고] |

### Out-of-Scope (다음 버전)

- [제외 항목 1]: [제외 사유]
- [제외 항목 2]: [제외 사유]
```

#### 6. 비기능 요구사항

```markdown
## 6. 비기능 요구사항

- **성능**: [응답 시간, 처리량 등]
- **정합성**: [데이터 일관성 요구사항]
- **중복 방지**: [멱등성 요구사항]
- **보안**: [인증, 인가, 데이터 보호]
- **법/규제**: [개인정보보호법, 정보통신망법 등 해당 시]
```

해당하지 않는 항목은 생략 가능하나, 성능과 보안은 반드시 포함.

#### 7. 기술 고려사항

```markdown
## 7. 기술 고려사항

- [기술 제약 또는 설계 결정] -- **확인 필요**
- [확정된 기술 결정]: [결정 내용과 근거]
- [확정된 기술 결정]: [결정 내용과 근거]
```

논의 중인 사항은 "확인 필요" 표시, 확정 사항은 근거와 함께 기재.

#### 8. 성공 지표 (KPI)

```markdown
## 8. 성공 지표 (KPI)

| 지표 | 현재 (Baseline) | 목표 | 측정 방법 |
|------|----------------|------|----------|
| [지표 1] | [현재값 또는 TBD] | [목표값] | [측정 방법] |
```

#### 9. 실험 계획 (해당 시)

A/B 테스트가 필요한 경우에만 포함. 없으면 이 섹션을 생략하고 10번부터 번호를 매긴다.

#### 10. 이해관계자 & 의존성 (해당 시)

외부 팀 의존성이 없으면 생략 가능.

#### 11. 미결 사항 (Open Questions)

```markdown
## 11. 미결 사항 (Open Questions)

- [ ] [미결 사항 1] -- 담당: [담당자] -- 기한: [날짜]
- [x] [해결된 사항] -- [결정 내용]
```

각 미결 사항에 반드시 **담당자**와 **기한**을 명시한다.

#### 12. 변경 이력

```markdown
## 12. 변경 이력

| 버전 | 날짜 | 변경 내용 | 작성자 |
|------|------|----------|--------|
| v0.1 | [날짜] | 최초 작성 | [작성자] |
```

### ID 체계

- **US-XX**: User Story 고유 ID (US-01, US-02, ...)
- **AC-XX**: Acceptance Criteria 고유 ID

### Acceptance Criteria 상세 (선택)

기능 요구사항의 수용 기준을 더 상세히 기술해야 할 경우 별도 섹션으로 분리:

```markdown
### US-01의 Acceptance Criteria

#### AC-01: [검증 포인트]
- **Given**: [사전 조건]
- **When**: [행동/트리거]
- **Then**: [예상 결과]
```

## Phase 3: Reviewer — 품질 검증

PRD 초안 작성 완료 후, **prd-reviewer agent** (subagent_type="prd-reviewer", model=sonnet)를 실행한다.

Reviewer에게 전달할 프롬프트:

```
다음 PRD 초안의 품질을 검증하라. .claude/agents/prd-reviewer.md 정의를 따른다.

검증 기준:
1. 12개 필수 섹션 커버리지
2. P0/P1 우선순위 태깅 완전성
3. 비기능 요구사항의 구체성 (성능/보안 필수)
4. 미결 사항의 담당자/기한 명시
5. Out-of-Scope 명시 여부
6. 수용 기준의 검증 가능성 (Given/When/Then 변환 가능한가?)
7. 품질 레퍼런스 대비 수준

판정: PASS | NEEDS_IMPROVEMENT
NEEDS_IMPROVEMENT 시 구체적 수정 사항을 우선순위순으로 제시하라.
```

---

## Phase 4: 수정 및 완료

### Reviewer 판정이 NEEDS_IMPROVEMENT인 경우

1. Reviewer 피드백을 반영하여 PRD 수정
2. 수정 사항이 경미하면 (2개 이하) 직접 수정 후 완료
3. 수정 사항이 중대하면 (3개 이상) Reviewer 재실행 (최대 2회)

### 완료 후

```markdown
## PRD 작성 완료

위 PRD를 검토해주세요. 수정이 필요하면 피드백 주세요:

### 피드백 예시:
- "User Story 추가해줘: [새로운 시나리오]"
- "P0 기능에 [기능명] 추가해줘"
- "비기능 요구사항에 동시성 처리 추가해줘"
- "기술 고려사항에 [결정사항] 확정으로 바꿔줘"

### 다음 단계:
1. PRD 확정 후 `/create-research` 실행
2. Research 완료 후 `/create-plan` 실행
```

## 에러 핸들링

| 상황 | 전략 |
|------|------|
| 품질 레퍼런스 파일 읽기 실패 | 일반 기준으로 작성, 레퍼런스 미참조 명시 |
| 팀/모듈 감지 실패 | 사용자에게 팀/모듈명 확인 요청 |
| Reviewer agent 실패 | 1회 재시도, 재실패 시 Quick Checklist로 자체 검증 |
| 사용자 입력 모호 | 핵심 질문 3개 이내로 범위 좁히기 |

## Quick Checklist

- [ ] 12개 섹션 모두 작성됨 (해당 없는 조건부 섹션은 생략 가능)
- [ ] 기능 요구사항에 P0/P1 우선순위 태깅됨
- [ ] 비기능 요구사항에 성능/보안 포함됨
- [ ] 기술 고려사항에 확정/미확정 구분됨
- [ ] 미결 사항에 담당자/기한 명시됨
- [ ] 변경 이력 작성됨
- [ ] Out-of-Scope 명시됨
- [ ] Reviewer PASS 획득 (또는 피드백 반영 완료)

## 파일 저장

`{team-name}/{feature-name}/prd.md` 또는 `{team-name}/{feature-name}/PRD.md`로 저장한다. 기존 저장소의 네이밍 컨벤션을 따른다.
