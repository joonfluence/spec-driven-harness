# 평가 루브릭 (PRD / Plan 공통 원칙: 체크 가능한 항목만, 주관 없음)

## PRD 루브릭 (각 1점, 총 12점)

1. 12개 섹션 모두 존재 (조건부 섹션은 "해당없음" 명시도 인정)
2. P0/P1처럼 명시적 우선순위 태깅이 모든 기능 요구사항에 존재
3. Out-of-Scope 항목이 최소 1개 이상, 사유 포함
4. 수용 기준이 Given/When/Then 또는 그에 준하는 검증 가능한 형태
5. 비기능 요구사항에 성능(구체적 수치) 포함
6. 비기능 요구사항에 보안/권한 포함
7. 비기능 요구사항에 멱등성/정합성(중복 생성 방지 등 도메인 특화 리스크) 포함
8. 데이터/도메인 경계 케이스(예: 타임존, 월말 날짜, 삭제 시 기존 인스턴스 처리) 최소 1개 이상 명시
9. 미결 사항에 담당자 + 기한이 모두 명시
10. KPI/성공지표에 베이스라인+목표+측정방법 모두 존재
11. 변경 이력 섹션 존재
12. 전체적으로 "확인 필요"와 "확정"이 구분되어 있어 개발자가 추측 없이 읽을 수 있음

## Plan 루브릭 (각 1점, 총 10점)

1. 아키텍처 다이어그램(ASCII 등 시각 자료) 존재
2. 기각된 설계 대안이 최소 1개 이상, 채택 이유와 함께 명시 (straw man 아님 — 대안이 실제로 그럴듯한가)
3. 데이터 모델의 상태 전이(정지/재개/건너뛰기/삭제 등)가 다이어그램 또는 표로 명시
4. API 설계에 Request/Response 예시 포함
5. Task별 고유 ID + 코드 스니펫 또는 구체적 구현 힌트 존재
6. Task 간 의존성/실행순서 명시
7. Risk & Mitigation에 도메인 특화 리스크(멱등성, 타임존, DST, 월말 날짜) 최소 2개 이상
8. PRD의 모든 P0 기능이 최소 1개 Task에 매핑됨 (AC Coverage)
9. 롤백/배포 전략 언급
10. 동시성/중복 실행(스케줄러 재시도, 중복 생성) 대응이 설계 수준에서 구체적으로 명시

## 채점 방식

- 각 항목 충족(1) / 부분충족(0.5) / 미충족(0)
- Baseline과 Harness(최종본) 각각 채점, 근거 인용(원문 발췌) 포함
- 총점 비교 + "Harness에만 있고 Baseline에는 없는 항목" 리스트를 핵심 증거로 제시
