# PRD: Recurring Task (반복 일정) 기능

- **문서 상태**: Final Draft v1.0
- **작성일**: 2026-07-01
- **작성자**: PM/Tech Lead
- **대상 제품**: 협업 태스크 관리 SaaS (Trello/Asana 유사 서비스)
- **관련 도메인**: Task, Project, Team, Notification (기존 존재), Calendar 연동 (기존 존재 가정)

---

## 1. Problem Statement

현재 서비스는 1회성 태스크 생성만 지원한다. 그러나 실제 팀 운영에서는 "매주 월요일 스프린트 플래닝", "매월 말일 정산 보고", "매일 스탠드업 체크"처럼 주기적으로 반복되는 업무가 전체 태스크의 상당 비중을 차지한다. 사용자는 이를 위해 매번 수동으로 태스크를 복제하거나, 캘린더/리마인더 등 외부 도구를 병행 사용하고 있다. 이는 다음과 같은 문제를 유발한다.

- **반복 생성 비용**: 사용자가 동일한 태스크를 손으로 재생성 → 실수(담당자 누락, 상세 정보 불일치) 및 시간 낭비.
- **가시성 단절**: 반복 업무가 시스템에 "일정"으로 등록되지 않아 팀 캘린더/워크로드 관점에서 예측이 안 됨.
- **경쟁 제품 대비 기능 격차**: Asana, ClickUp, Monday 등 주요 경쟁 제품은 이미 반복 태스크를 지원. 온보딩/전환 검토 시 이 기능의 부재가 실격 사유로 언급된 사례 다수(세일즈팀 피드백).

## 2. Goals & Non-Goals

### Goals (이번 릴리즈 범위)
1. 태스크 생성/수정 시 반복 규칙(매일 / 매주-요일지정 / 매월-날짜지정)을 설정할 수 있다.
2. 반복 규칙에 따라 다음 인스턴스가 정해진 시점에 자동 생성된다.
3. 사용자는 반복을 일시정지(pause) / 재개(resume)할 수 있다.
4. 특정 회차만 건너뛰기(skip single occurrence)가 가능하다.
5. 반복 규칙 자체(주기, 담당자, 제목/설명 템플릿 등)를 수정하거나, 반복 전체를 삭제할 수 있다.
6. 생성된 태스크 인스턴스는 기존 팀 캘린더 뷰에 노출되고, 기존 알림 파이프라인을 통해 마감/생성 알림을 받는다.
7. 반복 태스크임을 UI 상에서 명확히 식별할 수 있다 (아이콘/배지 + "다음 반복: n월 n일" 정보).

### Non-Goals (이번 릴리즈 제외, 후속 검토)
- 복잡한 반복 규칙(예: "매월 두 번째 화요일", "격주", "매 N일마다", RRULE 완전 지원) — v2 후보.
- 반복 태스크 간 종속성(예: 이전 회차 미완료 시 다음 회차 생성 보류).
- 반복 인스턴스의 서브태스크/체크리스트 템플릿 상속 고도화(1차는 단순 복제만 지원).
- 타 서비스(Google Calendar, Outlook) 양방향 동기화. 내부 팀 캘린더 연동만 포함.
- 반복 규칙 변경 이력 관리/감사로그 UI (백엔드 이벤트 로깅은 포함하되 별도 UI는 제외).
- 타임존이 서로 다른 팀원 간 "발생 기준 타임존" 개인화. 1차는 프로젝트 단일 타임존 기준.

## 3. Target Users & Use Cases

| 페르소나 | 시나리오 |
|---|---|
| 팀 리드 | 매주 월요일 오전 스프린트 플래닝 태스크를 자동 생성해 팀원에게 할당 |
| 오퍼레이션 매니저 | 매월 마지막 영업일에 "월간 정산 보고서 작성" 태스크 자동 생성 |
| 개인 기여자 | 매일 반복되는 "일일 체크리스트 점검" 개인 루틴 관리, 휴가 중엔 일시정지 |
| PM | 반복 회의 준비 태스크 중 공휴일이 낀 특정 회차만 건너뛰기 |

## 4. Functional Requirements

### 4.1 반복 규칙 생성 (P0)
- 태스크 생성/수정 폼에 "반복" 옵션 추가.
- 지원 주기:
  - **매일 (Daily)**: 매 N일 없이 1일 간격 고정 (v1은 interval=1만 지원, UI는 확장 여지 고려해 "매일" 고정 문구 사용 가능).
  - **매주 (Weekly)**: 요일 다중 선택 가능 (예: 월/수/금).
  - **매월 (Monthly)**: 날짜 지정 (예: "매월 15일"). 말일 지정(예: "매월 마지막 날") 지원. 31일처럼 해당 월에 없는 날짜는 "그 달의 마지막 날"로 보정(정책은 §7 참조).
- 반복 시작일, 종료 조건 선택 가능:
  - 종료 없음(계속 반복)
  - 특정 날짜까지
  - 총 발생 횟수까지 (예: 10회)
- 반복 태스크의 담당자/우선순위/라벨/설명/체크리스트는 매 회차 인스턴스에 동일하게 복제되는 "템플릿" 역할을 함.
- 소속 프로젝트 변경 불가(반복 규칙 수정 시 프로젝트는 고정, 필요 시 규칙 삭제 후 재생성).

### 4.2 인스턴스 자동 생성 (P0)
- 각 반복 규칙에 대해 다음 발생 시점(next occurrence)의 태스크 인스턴스를 사전 생성(lead time 설정, 기본 "발생일 당일 00:00 프로젝트 타임존 기준" 생성, lead time은 프로젝트 설정으로 추후 조정 가능하도록 확장 여지 설계).
- 생성된 인스턴스는 일반 태스크와 동일한 CRUD, 코멘트, 첨부파일, 서브태스크 기능을 모두 사용 가능.
- 각 인스턴스는 원본 반복 규칙(RecurringSeries)에 대한 참조를 가지며 "n번째 회차" 메타데이터를 보유.
- 이전 회차가 미완료 상태여도 다음 회차는 예정대로 생성됨 (v1은 종속성 없음 — Non-Goal 참조).
- 동일 시점에 대해 중복 인스턴스가 생성되지 않도록 멱등성 보장(스케줄러 재시도/중복 실행 대비).

### 4.3 일시정지 / 재개 (P0)
- 반복 규칙 단위로 Pause 가능. Pause 중에는 신규 인스턴스가 생성되지 않음.
- 이미 생성된 인스턴스는 Pause와 무관하게 그대로 유지(삭제되지 않음).
- Resume 시 "다음 발생 시점"부터 재개. Pause 기간 동안 놓친 회차를 소급 생성하지 않음(정책은 §7).
- Pause/Resume 이력은 감사 목적으로 백엔드에 기록.

### 4.4 특정 회차 건너뛰기 (Skip) (P0)
- 캘린더/반복 상세 뷰에서 향후 특정 회차를 선택해 "이번 회차 건너뛰기" 가능.
- Skip된 회차는 인스턴스를 생성하지 않고 "건너뜀"으로 표시되어 이력에 남음(캘린더에는 취소선/회색 처리로 표시, 완전히 숨기지 않음 — 팀원이 "왜 이번 주는 없지"라고 혼란스러워하지 않도록).
- 이미 생성된 인스턴스에 대해서도 "이 인스턴스 삭제 + 향후 이 회차는 건너뛴 것으로 기록" 액션 제공.

### 4.5 반복 규칙 수정 (P0/P1)
- 규칙(주기, 요일/날짜, 종료조건, 담당자, 라벨 등) 수정 가능.
- 수정 시점 기준으로 **"이번 회차만"** 또는 **"이후 모든 회차"** 적용 범위 선택 UI 제공 (Asana/Google Calendar와 동일한 UX 패턴). (P1: "이번 회차만" 옵션은 v1.1로 미루고 v1은 "이후 모든 회차 적용"만 지원 — 범위 결정은 §7 Open Question 참조)
- 이미 생성되어 완료/진행 중인 과거 인스턴스는 소급 변경되지 않음.

### 4.6 반복 규칙 삭제 (P0)
- 규칙 삭제 시 확인 다이얼로그: "향후 생성될 반복만 중단" vs "생성된 모든 미완료 인스턴스도 함께 삭제". 완료된 과거 인스턴스는 삭제 대상에서 제외(이력 보존).
- 소프트 삭제(deleted_at) 적용, 하드 삭제는 데이터 보존 정책에 따름(기존 삭제 정책 재사용).

### 4.7 캘린더 연동 (P0)
- 반복 규칙의 향후 발생 예정 시점(아직 인스턴스가 생성되지 않은 미래 회차 포함)을 팀 캘린더에 "예정된 반복" 형태로 표시.
- 이미 인스턴스화된 태스크는 기존 캘린더 표시 로직 그대로 사용(마감일 기준).
- 캘린더에서 반복 태스크는 반복 아이콘으로 구분 표시.

### 4.8 알림 연동 (P0)
- 인스턴스 생성 시 기존 "태스크 할당" 알림 트리거 재사용 (담당자에게 신규 태스크 알림).
- 반복 규칙이 Pause/Resume/삭제될 때 규칙 소유자 및 담당자에게 알림(옵션, 기본 On).
- 기존 마감 임박 알림(due-soon reminder)은 인스턴스가 일반 태스크이므로 별도 개발 없이 그대로 적용됨.

### 4.9 UI/UX 요구사항
- 태스크 상세 화면에 "반복" 섹션: 규칙 요약("매주 월, 수, 금마다 반복"), 다음 발생일, Pause/Resume 토글, 규칙 수정/삭제 진입점.
- 반복 태스크 카드에는 반복 배지 아이콘 표시.
- 반복 규칙 생성 폼은 생성 모달에서 "고급 옵션 펼치기" 형태로 노출(기본은 접힘 상태 — 대다수 태스크는 1회성이므로 기본 폼 복잡도를 높이지 않음).

## 5. Non-Functional Requirements

- **정확성**: 스케줄된 시각으로부터 인스턴스 생성까지 지연은 시스템 정의 SLA(예: 15분) 이내. 최소한 "당일 중" 생성 보장.
- **멱등성/중복 방지**: 동일 회차에 대해 정확히 1개의 인스턴스만 생성됨을 보장(분산 환경, 스케줄러 재실행 고려).
- **확장성**: 반복 규칙 수가 프로젝트/팀 규모에 비례해 증가해도 배치 처리가 선형적으로 확장 가능해야 함 (대량 반복 규칙 보유 대형 테넌트 고려).
- **감사 가능성**: 반복 규칙 생성/수정/삭제/Pause/Resume/Skip 이벤트는 감사 로그에 기록.
- **하위 호환성**: 기존 태스크/프로젝트 API는 변경 없이 동작. 반복 관련 필드는 옵셔널 확장으로 추가.
- **타임존**: 반복 발생 시점 계산은 프로젝트에 설정된 타임존 기준(기존 프로젝트 설정 재사용). DST 전환 주간에도 발생 요일/날짜가 어긋나지 않아야 함.

| 중복 생성 인시던트 | 0건 (Critical bug bar) |
| 지원 문의 감소 | "반복 일정 어떻게 만들어요" 관련 CS 문의 대비, "매번 태스크 복제 번거롭다" 유형 문의 50% 감소 |
| NPS/CSAT 코멘트 | 반복 태스크 관련 긍정 피드백 정성 트래킹 |

## 8. Rollout Plan

- **Phase 0**: 내부 dogfooding (자사 워크스페이스 한정), 기능 플래그로 제어.
- **Phase 1**: 베타 워크스페이스 opt-in (요청 기반), 모니터링 강화(중복 생성/누락 알림 대시보드).
- **Phase 2**: 전체 GA, 기능 플래그 제거.
- **Kill switch**: 인시던트 발생 시 스케줄러 배치만 즉시 중단 가능한 운영 토글 확보(기존 인스턴스/규칙 데이터에는 영향 없음).

## 9. Dependencies

- 기존 Task/Project/Team 도메인 모델 및 API.
- 기존 Notification 파이프라인(할당 알림, 마감 임박 알림).
- 기존 팀 캘린더 뷰/서비스.
- 백그라운드 잡 스케줄러 인프라(기존에 유사 배치 작업이 있다면 재사용, 없다면 PLAN.md에서 신규 구축 범위로 다룸).
